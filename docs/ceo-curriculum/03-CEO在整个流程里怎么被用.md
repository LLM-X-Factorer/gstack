# CEO 在 gstack 整个流程里怎么被用

> 前两份文档教你 **CEO 是谁、CEO 怎么想**。
> 这一份告诉你 **CEO 什么时候出场、和谁配合、留下的东西被谁拿走**。
>
> 用一句话讲：**CEO 不是一个人独唱，CEO 是整部交响乐的第一小提琴。** 它先发声、定调子，后面的设计师、工程师、发货员才跟着进来。

---

## 第一幕：CEO 是被怎么"叫醒"的？

CEO 不会自己跳出来。它有五个被叫醒的入口：

### 入口①：你直接喊它（最常见）

你打开 Claude Code，敲：

```
/plan-ceo-review
```

这是最直接的方式。你已经有一份计划，想让一个挑剔的人帮你看一遍。

### 入口②：你用 `/autoplan` 全套流水线

```
/autoplan
```

这是 CEO 出场最隆重的方式。它不是单独跑 CEO，而是把 CEO、设计师、工程师按顺序串成一条流水线：

```
CEO 评审 → 设计师评审 → 工程师评审 → DX（开发体验）评审
```

CEO 永远是**第一个**。原因很简单：如果连"这是不是要解决的问题"都没想清楚，让设计师琢磨界面、工程师琢磨架构，都是浪费。

📎 源头：`autoplan/SKILL.md.tmpl:107` —— "CEO → Design → Eng → DX 严格顺序"
📎 加载位置：`autoplan/SKILL.md.tmpl:270-272` 显式 load `plan-ceo-review/SKILL.md`

### 入口③：别的 skill 在中途**指路**给它

有几个 skill 自己跑着跑着，会跟你说"这事你该去找 CEO 谈谈"：

- **`/investigate`**（排查 bug 时）—— 如果发现这不是一个 bug，而是一个**范围**问题，它会把你引到 CEO（`investigate/SKILL.md:263`）
- **`/retro`**（回顾时）—— 复盘完一件事，发现有更大的策略问题，引到 CEO（`retro/SKILL.md:241`）
- **`/office-hours`**（产品脑暴时）—— 产出一份"设计文档"之后，下一步就是 CEO

### 入口④：发货流程**展示**它

`/ship`（准备发货时）会在仪表板上**显示** CEO 评审有没有跑过：

> "CEO Review not run — 产品/业务变更建议跑一下"

但请注意：它**不强制**。bug 修复、重构、基础设施变更可以跳过。产品/业务功能变更建议补上。

📎 源头：`ship/SKILL.md:227,847,873,896`

### 入口⑤：合并部署时**回读**它的笔记

`/land-and-deploy`（合并 + 部署 + 看 canary）跑的时候，会去读 CEO 留下的笔记——什么时候评审过、当时是什么模式、什么状态——拼到一张大表里。

📎 源头：`land-and-deploy/SKILL.md.tmpl:390`

## 第二幕：CEO 在流水线里**坐**在哪个位置？

```
你（用户）
   │
   ▼
┌──────────────────────────────────────────┐
│   Phase 0.5: Codex 登录 + 版本检查        │  ← 流水线开场
└──────────────────────────────────────────┘
   │
   ▼
┌──────────────────────────────────────────┐
│   Phase 1: CEO 评审                       │  ← 我们现在在讲的
│   - 问题对不对？                          │
│   - 现有代码能不能复用？                  │
│   - 12 个月后的理想状态长啥样？           │
│   - 2-3 种实现方案对比                    │
│   - 选 mode（扩张 / 精选 / 守住 / 砍掉）  │
│   - 11 个评审维度                         │
└──────────────────────────────────────────┘
   │  把"CEO 关键问题和分歧"传下去
   ▼
┌──────────────────────────────────────────┐
│   Phase 2: Design 评审                    │  ← 拿到 CEO 的 scope 决议，看 UX
└──────────────────────────────────────────┘
   │
   ▼
┌──────────────────────────────────────────┐
│   Phase 3: Eng 评审                       │  ← 拿到前两个的结论，看架构
└──────────────────────────────────────────┘
   │
   ▼
┌──────────────────────────────────────────┐
│   Phase 4: DX 评审（可选）                │
└──────────────────────────────────────────┘
   │
   ▼
计划修订完成 → 进入实现
```

**为什么是这个顺序？**

因为这是个**漏斗**。CEO 先把范围定了——这一步如果错了，后面再细致也是给错的方向加细节。设计师在 CEO 定下来的范围里挑 UX。工程师在设计师定下来的 UX 里挑架构。每一层只解决自己那一层的问题，不偷别人的活。

📎 源头：`autoplan/SKILL.md.tmpl:415-416` —— Phase 1 的"关键问题和分歧"会传给 Phase 2 的设计 voice

## 第三幕：CEO 拿着什么**进**门，又留下什么**出**门？

### 🎒 CEO 进门时手里拿的东西

CEO 不是凭空开始评审的。它会**先把现场扫一遍**：

1. **读 `CLAUDE.md`** —— 项目的规矩、命令、约定
2. **读 `TODOS.md`** —— 目前还没做的事
3. **看 `git log` 和 `git diff`** —— 最近发生了什么、这次想改什么
4. **找设计文档** —— 在 `~/.gstack/projects/{项目名}/*-design-*.md` 里找有没有相关设计稿
5. **找交接笔记** —— 在同样位置找 `*-{分支}-ceo-handoff-*.md`，看上次 CEO 评审给后续留了什么话

📎 源头：`plan-ceo-review/SKILL.md.tmpl:124-162` —— "PRE-REVIEW SYSTEM AUDIT"

> 这一步很关键。一个不读现场就开口的 CEO，是个**坏** CEO。它只会复读教科书。
> 一个先看一圈再开口的 CEO，才能说出"等等，三个月前你已经做过类似的事，那次你后悔了，要不要先聊聊那次为什么后悔"。

### 📤 CEO 出门时留下的东西

CEO 评审完，会留下**两份产物**：

#### 产物①：CEO 计划文件（详细版）

写到：

```
~/.gstack/projects/{项目名}/ceo-plans/{日期}-{功能名}.md
```

里面长这样：

- 0A 的"前提挑战"清单
- 0B 的"现有代码能复用什么"映射表
- 0C 的"现在 → 这个计划 → 12 个月理想"三段图
- 0C-bis 的"2-3 种实现方案"对比表（小/中/大/特大 effort）
- 0D 的 mode 决议 + 每一项 expansion 用户选了哪个
- 0E 的"实现的每一小时会撞到什么坑"预言
- 0F 的最终 mode 确认
- 接下来 11 个评审维度的逐项结论

📎 源头：`plan-ceo-review/SKILL.md.tmpl:320-361`

#### 产物②：JSON 元数据（速查版）

通过 `gstack-review-log` 工具把状态写到 GBrain（gstack 的记忆系统），格式是结构化的：

```json
{
  "status": "DONE / DONE_WITH_CONCERNS / BLOCKED",
  "mode": "EXPANSION / SELECTIVE / HOLD / REDUCTION",
  "scope_in": [...],
  "scope_deferred": [...],
  "scope_excluded": [...]
}
```

📎 源头：`plan-ceo-review/SKILL.md.tmpl:800` 附近

这份 JSON 是给**机器**看的——`/ship`、`/land-and-deploy` 都会读它，用来在仪表板上显示"CEO 评审过没有、什么时候过的、当时什么模式"。

## 第四幕：CEO 和它的**同事们**怎么分工？

gstack 里有四个"评审型"的 skill，它们经常被搞混。这张表帮你分清楚：

| Skill | 出场时机 | 评审什么 | 一句话总结 |
|-------|---------|---------|-----------|
| **`/office-hours`** | 你还**没有**计划，只有"想法" | 这想法值不值得做？用户真有这个痛点吗？ | "先别动手，先想清楚要做什么" |
| **`/plan-ceo-review`** | 你**有**计划了，但还没动手 | 范围对不对？前提对不对？目标对不对？ | "等等，我们先重塑这件事" |
| **`/plan-design-review`** | 计划里有 UI / UX | 信息层级、交互状态、空状态、加载态 | "用户实际怎么用这个界面？" |
| **`/plan-eng-review`** | 计划即将进入实现 | 架构、数据流、错误处理、性能、测试 | "代码层面有几个地雷？" |

**口诀：**

- 还**没有**想法 → office-hours（脑暴）
- 有了想法、**没有**计划 → office-hours（脑暴 + 出设计文档）
- 有了计划、**没**评审 → plan-ceo-review（先来 CEO）
- CEO 过了、**触及 UI** → plan-design-review
- CEO 和设计都过了、**要写代码了** → plan-eng-review
- 想**一键串起来** → autoplan

**特别注意：** office-hours 是**在 CEO 之前**。如果你连"要不要做"都不确定，先 office-hours，再 CEO。CEO 不是用来回答"我该不该做这个产品"的——CEO 是用来回答"这个我已经决定要做的事，应该以什么形态被做出来"的。

📎 源头：`office-hours/SKILL.md.tmpl:16` 显式说自己用在 CEO 评审**之前**

## 第五幕：CEO 留下的笔记被**谁**拿去读了？

CEO 不是写完笔记就关门走人。它的笔记会被三个地方反复**读**：

### 读者①：autoplan 的下游 Phase

Phase 2（设计师）和 Phase 3（工程师）会**继承** CEO 的关键发现——尤其是"分歧"和"未解决的问题"。这意味着 CEO 不需要在笔记里写满所有细节，只需要把**有争议**的点写清楚，让下游知道哪里需要继续追问。

📎 源头：`autoplan/SKILL.md.tmpl:415-416`

### 读者②：`/ship`（发货前）

发货前的仪表板会显示一行：

> "CEO Review: ✅ 2026-05-10 — HOLD SCOPE — DONE"

或者：

> "CEO Review: ⚠️ Not run — 这是产品功能变更，建议补一下"

它**不会**因为没跑 CEO 就拒绝发货——但它会**提醒**你这是不是一个该跑 CEO 的变更。

📎 源头：`ship/SKILL.md:847,896`

### 读者③：`/land-and-deploy`（合并部署时）

部署流程会扫一遍所有评审记录（CEO、设计、工程），拼成一张"这个变更经过了哪些门、状态是什么"的速查表。这张表跟着 PR 一起，帮 reviewer 一眼看清"这个 PR 的质量门走全了没有"。

📎 源头：`land-and-deploy/SKILL.md.tmpl:390` —— "解析 plan-ceo-review 的时间戳和状态"

### 读者④：未来的你 / 未来的另一个 Claude

CEO 笔记**留在仓库里**（`~/.gstack/projects/{项目名}/ceo-plans/`）。三个月后你回到这个项目，问"当初为什么这个功能做成这样、当时考虑过哪些方案、为什么砍了 X"——答案就在这些笔记里。

这是 CEO 的**长尾价值**：它不只是当下决策的工具，它是项目的**机构记忆**。

## 第六幕：什么场景该召唤 CEO？速查图

```
你手上的东西是？
│
├─ 一个模糊的"我想做点什么"      → /office-hours（先脑暴）
│
├─ 一份明确的"我打算做 XYZ" 文档
│   │
│   ├─ 想要全套评审（最稳）        → /autoplan
│   │
│   ├─ 只想策略层面被挑战          → /plan-ceo-review
│   │
│   ├─ 只想看架构是否合理          → /plan-eng-review
│   │
│   └─ 只想看 UX 是否到位          → /plan-design-review
│
├─ 一个 bug / 报错                 → /investigate
│   （investigate 如果发现是策略问题，会自己引你到 CEO）
│
├─ 一个已经写完的 PR               → /review
│
├─ 已经完成、想复盘                → /retro
│
└─ 已经评审、准备发货              → /ship
    （ship 会展示 CEO 评审状态，但不强制）
```

## 第七幕：给你的两个小练习

**练习 1：**
你想给一个开源项目加一个"导出 CSV"功能。你已经知道用户要这个。你不确定应该怎么实现。该召唤谁？

> 答案：**`/plan-ceo-review`**。需求验证已经做完（office-hours 不需要了），但你不确定**实现形态**——是按钮触发？是 API？是命令行？是后台任务？CEO 的 0C-bis（实现方案对比）正是回答这个的。

**练习 2：**
你的产品在生产环境报了一个 NullPointer，你不知道原因。该召唤谁？

> 答案：**`/investigate`**。这不是范围问题，是事故问题。investigate 会带你逐步还原现场。**除非**它发现这是一个"我们的整套数据流根本不该长这样"的策略问题——那时候它会主动建议你切到 CEO。

---

## 一句话收尾

> **CEO 是流程的起点，不是终点。**
>
> 它的工作不是给你"通过 / 不通过"的盖章，而是在你动手之前，把"这件事到底是什么、值不值得做、怎么做最划算"问到底。
>
> 流程后面的每一个角色——设计师、工程师、发货员——都站在 CEO 给他们划好的地图上工作。
>
> 所以：**不要跳过 CEO**。跳过 CEO，等于让一群熟练工人在没有地图的森林里开工。
